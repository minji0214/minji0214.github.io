# AI agent에도 디자인 패턴이 있다는 사실을 아시나요?

- Agent design patterns는 **요청의 복잡도 / 불확실성 / 실행 제약(시간·비용·품질)**에 맞춰 에이전트를 설계하기 위한 대표적인 오케스트레이션(orchestration) 패턴들이다.
- 하나의 거대한 프롬프트로 모든 걸 해결하려 하기보다, **역할을 나누고(전문화) 흐름을 설계**해 더 예측 가능하고 유지보수 가능한 시스템을 만든다.

## 왜 AI agent 디자인 패턴을 알아야 하나

- **“프롬프트만 잘 쓰면 되지 않나?”가 자주 깨진다**: 처음엔 간단한데, 요구가 하나씩 붙으면(조건 추가, 형식, 예외 처리) 금방 복잡해진다. 이때 패턴은 “이건 한 번에 말로 해결할 문제가 아니라, 단계로 나눠야겠다”를 판단하게 해준다.
- **실무에서 제일 흔한 문제: 결과가 들쭉날쭉하다**: 같은 질문을 해도 답이 매번 다르면 서비스에 넣기 어렵다. 그래서 “초안 → 체크(규칙/테스트/근거 확인) → 수정” 같은 **검수 루프**를 구조로 만들어 품질을 안정화한다.
- **시간이 늘 부족하다(그리고 토큰도 돈이다)**: 무조건 길게 돌리면 정확해질 것 같지만, 실제로는 “필요한 것만” 돌리는 게 중요하다. 패턴을 알면
    - 급한 답은 빠르게(단일)
    - 리서치/정리는 병렬로(병렬)
    - 최종 답만 검수(루프)
    
    처럼 **속도와 품질의 균형**을 잡기 쉬워진다.
    
- **에러가 났을 때 어디서 깨졌는지 찾기 쉽다**: 한 덩어리 프롬프트는 “왜 망했지?”가 막막하다. 단계가 분리돼 있으면 “검색이 잘못됐나 / 요약이 틀렸나 / 검증이 빠졌나”처럼 **원인 위치가 보인다.**
- **팀 작업이 된다(나중에 내가 나를 돕는다)**: ‘리서치 담당’, ‘작성 담당’, ‘검증 담당’처럼 역할을 나누면, 특정 단계만 바꿔 끼우거나 개선하기가 쉽다. 한 번 만들어두면 다음 업무에도 재사용된다.
- **현실에서는 ‘모르는 척’이 아니라 ‘확인’이 필요하다**: 에이전트는 그럴듯하게 말할 수 있지만, 우리는 실제 데이터(검색/DB/로그/테스트)로 확인해야 한다. 패턴은 “언제 툴로 확인할지, 실패하면 어떻게 대체할지”까지 포함해 더 현실적인 흐름을 만든다.

## Pattern 1. Single agent

- 하나의 에이전트가 **계획 → 툴 호출 → 결과 조합**까지 전부 수행한다.
- 예시
    - “샌프란시스코 여행 계획 세워줘”처럼 **단일 목표**
    - “늦게 여는 일식집 찾아서, 거기까지 가장 빠른 길까지 안내해줘”처럼 **탐색 + 경로 안내**가 한 번에 섞이는 경우(요구가 커질수록 프롬프트/추론이 급격히 커짐)
- 장점
    - 구현이 단순하고 디버깅이 쉽다.
    - 호출 수가 적어 **latency/비용이 낮을 수 있다.**
- 단점
    - 요구사항이 커지면 시스템 프롬프트가 비대해지고, 결과 품질을 통제하기 어렵다.
    - 컴포넌트 재사용이 어렵고 **single point of failure**가 되기 쉽다.

## Pattern 2. Sequential agent (Pipeline)

- 여러 서브 에이전트를 **정해진 순서대로** 실행한다. (multi-agent / pipeline)
- 예시
    - (1) 식당 찾는 에이전트 → (2) 교통/경로 에이전트 → (3) 최종 요약/결정 에이전트
- 장점
    - 흐름이 명확해 **예측 가능한 실행**이 가능하다.
    - 테스트/디버깅이 쉽다.
    - “필요한 단계만” 설계할 수 있어 비용 관리가 쉽다.
- 단점
    - 설계된 단계 밖의 상황에 유연하지 못하다(스텝 스킵/재계획이 어렵다).
    - **누적 latency**: B는 A가 끝날 때까지 기다려야 한다.

## Pattern 3. Parallel agent

- 서로 독립적인 서브 에이전트를 **동시에** 실행하고, 결과를 합친다.
- 예시
    - 식당 후보 탐색(에이전트 A) + 이동 수단/소요 시간 탐색(에이전트 B) 를 병렬로 진행 후, 합쳐서 추천
- 장점
    - 전체 latency 감소(동시에 수행).
    - sequential의 “예측 가능성”을 어느 정도 유지하면서 속도를 개선할 수 있다.
- 단점
    - 의존성/상태 관리가 어렵다(경쟁 조건, 결과 불일치).
    - 디버깅 난이도가 증가한다.
    - 컴퓨팅 리소스 contention 가능.

![image.png](../../contentsImgs/ai-agent-design-system-3.png)

## Pattern 4. Loop pattern (Review & Critique)

- 하나의 에이전트(또는 팀)가 **평가 → 수정 → 재시도** 같은 루프를 돈다.
- 예시
    - 답변 초안 생성 → 규칙/테스트로 검증 → 실패 시 다시 생성/수정
    - 코드 생성 → 빌드/테스트 실행 → 에러 로그 기반 수정 반복
- 장점
    - 단발성 추론보다 품질/정확도를 끌어올리기 좋다.
    - 결과물이 특정 품질 기준과 제약을 만족하도록 만들기 쉽다.
    - “정답이 1번에 나오기 어렵다” 유형(코드/리서치/정리)에 강하다.
- 단점
    - 반복 횟수만큼 비용/시간이 늘어날 수 있다(무한 루프 방지 장치 필요).
    - 평가 기준(validator) 설계가 중요하다.

## Pattern 5. Coordinator (Router) pattern

- **상위(감독) 에이전트**가 목표를 쪼개고, 하위 에이전트/팀에 작업을 할당한 뒤 결과를 취합한다.
- 언제 쓰나
    - 문제를 어떤 단계로 풀지 자체가 불명확하거나, 작업 종류가 다양해서 “고정된 파이프라인”이 잘 맞지 않을 때
- 장점
    - 작업 분해/할당을 통해 확장성이 높다.
    - 하위 에이전트를 더 작고 전문적으로 유지할 수 있다.
    - 런타임에 동적으로 라우팅(dynamic routing)할 수 있다.
- 단점
    - 오케스트레이션(감독)의 품질이 전체 품질을 좌우한다.
    - 상태/메모리 공유, 평가/선택 로직까지 고려해야 해서 설계 복잡도가 높다.
    - 라우팅/취합을 위한 추가 호출로 latency & cost가 늘 수 있다.

## Pattern 6. Agent as Tool

- LLM을 “뇌”로 두고, 필요할 때 **툴(검색/DB/코드 실행 등)**을 호출해 환경과 상호작용하는 형태
- Single agent의 구현 형태로도 볼 수 있고, multi-agent의 각 노드가 LLM agent일 수도 있다.
- 포인트
    - “모델의 추론” + “외부 세계의 사실(툴 결과)”을 결합해야 성능이 올라간다.
    - 툴 결과의 신뢰성/권한/실패 처리(재시도·fallback)까지 설계 범위에 포함된다.

## Pattern 5 vs 6: Coordinator(Router) vs Agent as Tool

Pattern 5와 6은 둘 다 “단일 에이전트가 모든 걸 다 한다”에서 벗어나 **구조적으로 복잡도를 낮추는 방식**이지만, 설계의 초점이 다르다.

### 핵심 차이(한 문장)

- **Pattern 5(Coordinator/Router)**: “어떤 작업을 누구에게 시킬지”를 결정하고 여러 작업 결과를 **조율·취합**하는 상위 오케스트레이터가 중심
- **Pattern 6(Agent as Tool)**: “추론(LLM)”이 중심이고, 부족한 정보를 채우기 위해 검색/DB/코드 실행 같은 **외부 툴을 호출**해 사실을 가져오는 방식

### 언제 5번이 더 적합한가

- 요청을 처리하는 데 필요한 작업 종류가 다양해서, 매번 단계가 달라지는 경우
    - 예: (1) 리서치, (2) 코드 생성, (3) 문서 요약, (4) 정책 체크처럼 task type이 섞임
- 도메인별로 강점이 다른 에이전트(또는 프롬프트/모델)를 **전문화**시켜두고, 상황에 맞게 조합하고 싶을 때
- “실행 그래프” 자체가 동적이라 고정 파이프라인(Sequential)이 잘 맞지 않을 때

### 언제 6번이 더 적합한가

- 문제의 중심이 “작업 분배”가 아니라, **정확한 외부 사실/상태**를 가져와야 풀리는 경우
    - 예: 최신 문서 검색, 데이터베이스 조회, 에러 로그 기반 수정, API 상태 확인
- 툴 결과(ground truth)를 기반으로 **추론을 보정**해야 할 때(환각 방지/정확도 향상)
- 한 에이전트가 충분히 판단할 수 있지만, 필요한 입력이 툴 없이는 얻기 어려운 경우

### 설계 관점에서의 체크포인트

- Pattern 5의 난이도 포인트
    - 라우팅 품질(분해/할당)이 전체 품질을 좌우
    - 상태 공유(메모리), 결과 충돌 해결, 취합 기준(선택/평가 로직) 설계가 필요
- Pattern 6의 난이도 포인트
    - 툴 호출 실패/권한/재시도·fallback 설계가 품질을 좌우
    - 툴 결과를 “어떻게 신뢰하고 해석할지”에 대한 규칙이 필요

### 같이 쓰는 경우(실무에서 흔한 조합)

- 상위는 **Coordinator(5)**로 두고, 하위 워커 에이전트들은 각자 필요한 순간에 **Tool 호출(6)**을 하는 형태
    - 예: Coordinator가 “리서치/정리/검증” 워커에게 작업을 나누고, 리서치 워커는 검색 툴을, 검증 워커는 테스트 툴을 호출

## 패턴 선택 가이드

![image.png](../../contentsImgs/ai-agent-design-system-9.png)

- 단순/짧은 요청 → Single
- 단계가 고정된 업무 프로세스(예: ETL, 리포트 생성) → Sequential
- 서로 독립적인 조사/계산을 동시에 해야 함 → Parallel
- 작업 분해 자체가 동적이고 팀/역할이 다양함 → Coordinator/Hierarchical
- 품질을 끌어올리기 위해 반복 검증이 필요함 → Loop
